Locked by the planner — do NOT re-litigate or re-ask the user. These resolve CONTEXT.md's
"how the round cap is lifted" discretion item and 26-RESEARCH.md Pitfall 2 / Open Question 2:

- **The cap lift is a keyword-only `radio: bool = False` param on `try_auto_queue`.** Default
  False renders the path byte-identical — the standing additive-change discipline (Phase 14's
  optional `kind` clause, Phase 16's `pre_recalled_memories=None` bypass, Phase 21's
  `guild_scoped`). Not a wrapper, not a second refill core: a wrapper would duplicate the
  pipeline D-01 exists to reuse.
- **Radio does NOT touch `auto_queue_rounds` or `auto_queue_results` at all** (26-RESEARCH
  Pitfall 2's recommendation, adopted). When `radio=True`: the cap check is skipped, the
  round counter is not incremented, and the results dict is not reset. Post-radio auto-queue
  therefore resumes with exactly the counters it had before radio started — no stale state
  crossing the radio/non-radio boundary. `/radio start` and `/radio stop` both call
  `reset_auto_queue()` (26-05) so the boundary is clean in both directions.
- **The `if not recent: return` bail must not fire for a seeded radio.** `try_auto_queue`
  currently bails when there is no recent song history (`cogs/ai.py:296`) — which would make
  `/radio start "daft punk"` silently do nothing in a room with no history, defeating SC-1's
  headline case. The seed IS the context when history is absent. Change the condition to also
  require the absence of a radio seed; when `radio=False` it is byte-identical.
- **The whole announce block is suppressed while radio is armed, not just the ignored line.**
  D-05 explicitly locks suppression of the `AUTO_QUEUE_IGNORED` announce + the
  `auto_queue_ignored` memory write. The plain `AUTO_QUEUE_ANNOUNCE` line ("fine, since nobody
  else is stepping up, here's what i picked") is suppressed too because during radio it would
  post every ~3 tracks for hours, and its text is a lie during radio — somebody DID step up,
  they typed `/radio start`, which already announced. D-05's own framing ("a behavior
  regression in a feature meant to run unattended") is the warrant. The now-playing embed
  remains the visible surface for what radio is playing.
- **The `music_cog`/`queue` resolution block is hoisted** above the prompt build, because the
  seed and played-set live on `MusicQueue` and the prompt needs them. The only behavioural
  delta on the non-radio path: when `MusicCog` is absent, `try_auto_queue` now bails BEFORE
  spending a Gemini call instead of after — strictly an improvement, no test depends on it.
</planner_decisions>
