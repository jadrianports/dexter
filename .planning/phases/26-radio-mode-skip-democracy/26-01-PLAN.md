"Claude's / Planner's Discretion" items:

- **A3 — radio armed-state lives on `MusicQueue`, and `clear()` disarms it.** Rationale:
  D-11 makes radio and `loop_mode` mutually exclusive, which means radio is the *same
  category* of state as `loop_mode` — and `clear()` already resets `loop_mode` to `OFF`.
  Putting the disarm in `clear()` covers ALL FOUR existing teardown sites for free
  (`/stop` at `cogs/music.py:1731`, `stop_button`→`_do_stop` at `:1069`, `idle_check` at
  `bot.py:963`, reconnect-failure at `cogs/music.py:2204`) — every one of them already
  calls `clear()`. **This means Phase 26 needs ZERO changes to `bot.py`.** Human `/play`
  never calls `clear()`, so D-07's "a human `/play` only injects" also falls out for free.
  This is NOT the `auto_lyrics` precedent (a *server preference* deliberately surviving
  `clear()`) — radio is playback state, like `loop_mode`. The reset is an explicit,
  commented line in `clear()`, not reliance on object lifetime (D-08's stated spirit).
- **D-08 (not persisted) is structural:** `services/queue_persistence.py::persist` builds an
  explicit typed dict of named keys (`tracks`/`current_index`/`loop_mode`/`text_channel_id`/
  `active_filter`), never `__dict__`. New `MusicQueue` fields are therefore unpersisted by
  construction. Do NOT add radio keys to that payload.
- **A2 — `RADIO_LOOKAHEAD_DEPTH = 2`.** Phase 6's prefetch needs exactly 1 next track to
  exist; 2 leaves a full track (~3-4 min) of runway for a Gemini + YouTube round trip, plus
  one retry cycle if a priority-2 refill is rejected (D-04's "runway before dead air").
  1 leaves zero margin; 3+ refills more aggressively than needed on a shared 15 RPM budget.
- **No session played-set cap knob.** The armed-radio lifetime IS the bound (D-08: dies with
  radio; D-07: disarmed at every teardown). A 6-hour session accumulates ~90 video_ids
  (~1 KB). A cap would silently reintroduce the exact repeats D-03 exists to prevent, in
  exchange for bounding a structure that tops out around a kilobyte. The *prompt hint* IS
  capped (`RADIO_ALREADY_PLAYED_HINT_CAP`) — the hard filter stays uncapped. That split is
  the Phase 14 discipline exactly: the hint is advisory, the hard gate is the guarantee.
- **The played-set is `dict[str, str]` (video_id → "Title by Artist"), one field serving
  both roles.** D-03 says "track the video_ids ... feed them to the prompt as an already-played
  hint" — but raw video_ids are useless to Gemini. Keys drive the D-03 hard post-filter
  (membership); values drive the human-readable prompt hint. Insertion-ordered, so the hint
  is chronological and the cap takes the most recent N.
- **`should_refill_radio` includes a `humans_present` gate.** `_on_track_end` already computes
  it (`cogs/music.py:864`) for `decide_on_track_end`, so it is free. Refilling into an empty
  room burns the shared 15 RPM budget for nobody (T-26-02) — and it mirrors the
  `humans_present` gate `decide_on_track_end` already applies to AUTOQUEUE.
</planner_decisions>
