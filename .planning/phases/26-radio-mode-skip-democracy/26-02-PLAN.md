<planner_decisions>
Locked by the planner — do NOT re-litigate or re-ask the user:

- **A1 — `SKIP_VOTE_MAJORITY_RATIO = 0.5`, and the arithmetic is `floor(n * ratio) + 1`, NOT
  the `n // 2 + 1` shown in 26-RESEARCH.md / 26-PATTERNS.md.** Those sketches hardcode 0.5 and
  therefore ignore the knob — but D-09 explicitly says "the fraction is a `config.py` knob", so
  `n // 2 + 1` would make the knob a lie (setting 0.66 would change nothing). `floor(n*ratio)+1`
  is "strictly more than the ratio" — the honest generalisation of D-09c's "strictly more than
  half", and it reproduces D-09c's locked table exactly at ratio=0.5: 1->1, 2->2, 3->2, 4->3.
  **Clamp the result to `len(listener_ids)`** so a misconfigured ratio (e.g. 1.0) degrades to
  unanimity instead of making skipping mathematically impossible and wedging the queue forever.
- **`required_votes` is a separate exported pure function**, not inlined in `decide_skip`. The
  glue needs the same number for the D-18 tally's `{required}` slot, and the Phase 10 D-02 seam
  rule forbids the cog re-deriving it. One arithmetic source, two consumers.
- **Vote state home: `MusicQueue`, keyed by `(current_index, video_id)`, auto-resetting on
  read.** 26-RESEARCH OQ3 left this open. Hooking every `current_index` mutation site
  (`_advance`, `previous`, `jump_to`, `clear`) would be four places to keep in sync forever;
  keying the vote set to the track's identity makes the reset structural — any track change
  yields a different key, so the votes are empty by construction. `current_index` is part of
  the key (not just `video_id`) so the same song queued twice at different positions gets its
  own vote. `/replay` re-plays the SAME track at the SAME index, so votes correctly persist —
  D-17 says votes clear when the track CHANGES, and a replay is not a change.
- **`SkipVerdict` gate order is fixed: requester -> solo -> already-voted -> tally.** The
  requester bypass must precede the idempotency check so a requester who already voted still
  bypasses (they can always pull their own pick — D-13a's escape hatch from the D-09c
  duo-holdout).
- **`decide_skip` intersects the votes with `listener_ids` when counting** (`len(new_votes &
  listener_ids)` is NOT used — see below). Per D-17, a departed voter's vote STAYS COUNTED, so
  the tally is `len(new_votes)`, not the intersection. 26-PATTERNS.md's sketch uses
  `len(new_votes & listener_ids)`, which would silently DROP a departed voter's vote and is a
  direct D-17 violation ("a walkout can't strand an open vote below a threshold that just
  dropped"). Count `len(new_votes)`; use `listener_ids` ONLY as the live denominator.
</planner_decisions>
