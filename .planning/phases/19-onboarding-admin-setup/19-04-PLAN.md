<objective>
Ship the user-facing `/setup` admin surface (ONBOARD-02/03/04): a new `cogs/admin.py` with an
`app_commands.Group` carrying three subcommands — `/setup channel` (native typed-channel dropdown,

## Trust Boundaries

| Discord interaction → /setup write | an arbitrary guild member invokes a privileged config command; the permission gate + guild-derived id are the trust boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-19-01 | Elevation of Privilege | /setup permission gate | mitigate | Inline `interaction.permissions.manage_guild` check via `_require_guild_admin`, the FIRST statement of every subcommand, before any data access (mirrors cogs/ops.py:252). `default_permissions` on the Group is a UI hint only and is never the enforcement mechanism (verified finding #4) |
| T-19-03 | Elevation of Privilege / Tampering | confused-deputy guild targeting | mitigate (structural) | No `/setup` subcommand accepts a `guild`/`guild_id` argument; every write derives `guild_id` from `interaction.guild.id` (server-provided, un-spoofable within an interaction). Enforced by the command signatures, not a runtime check |
| T-19-05 | Tampering | D-06 send_messages check-then-write TOCTOU | accept (self-healing) | The check-then-write window is milliseconds; a permission revoked after `/setup channel` is caught by the strict resolver's next ambient send (D-03 silent-skip). No new code |
| T-19-SC | Tampering | package installs | mitigate | N/A — zero new dependencies this phase |
</threat_model>
