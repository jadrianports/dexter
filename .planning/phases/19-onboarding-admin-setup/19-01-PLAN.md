    ambient_roasts_enabled, vision_roasts_enabled` — and use it in every RETURNING clause below.

    1. `insert_guild_config_if_absent(pool, *, guild_id: str) -> asyncpg.Record | None` (D-14): one
       `INSERT INTO guild_config (guild_id) VALUES ($1) ON CONFLICT (guild_id) DO NOTHING RETURNING <full cols>`
       via `conn.fetchrow`. Returns the Record on a genuine insert, None on conflict. Writes ONLY
       `guild_id` — `configured` (false), `ambient_roasts_enabled` (true), `vision_roasts_enabled` (true)
       all fall through to their column defaults per D-10/D-20. Docstring must state it is the sibling of
       `seed_guild_config_if_absent` and does NOT set `configured=true`.
    2. `configure_guild_first_time(pool, *, guild_id: str, channel_id: str) -> asyncpg.Record | None`
       (D-19/D-20 first-configure): an UPSERT so it is robust whether or not the row already exists (under
       a fail-closed cache the `/setup channel` first-configure branch may run against a missing row) —
       `INSERT INTO guild_config (guild_id, ambient_channel_id, configured, vision_roasts_enabled)
       VALUES ($1, $2, true, false) ON CONFLICT (guild_id) DO UPDATE SET ambient_channel_id =
       EXCLUDED.ambient_channel_id, configured = true, vision_roasts_enabled = false, updated_at = now()
       RETURNING <full cols>`. DO UPDATE is correct here (unlike the env-seed's DO NOTHING) because this is
       a deliberate admin write. It is the one place vision is turned off (D-19); `ambient_roasts_enabled`
       is never named, so it stays at its existing value / true default (roasts ON). Docstring notes this
       asserts the default-vision-OFF policy over any pre-channel `/setup vision on` (rare; surfaced by the
       D-05 echo).
    3. `redesignate_guild_channel(pool, *, guild_id: str, channel_id: str) -> asyncpg.Record | None`
       (D-03): `UPDATE guild_config SET ambient_channel_id = $2, updated_at = now() WHERE guild_id = $1
       RETURNING <full cols>`. Touches ONLY the channel — never `configured`, never either toggle (D-20's
       "a re-designate must not reset a toggle the admin has since changed"). Only ever called when a row
       with `configured=true` already exists, so a plain UPDATE is correct.
    4. `set_ambient_roasts_enabled(pool, *, guild_id: str, enabled: bool) -> asyncpg.Record | None`:
       `UPDATE guild_config SET ambient_roasts_enabled = $2, updated_at = now() WHERE guild_id = $1 RETURNING <full cols>`.
    5. `set_vision_roasts_enabled(pool, *, guild_id: str, enabled: bool) -> asyncpg.Record | None`: same shape,
       `vision_roasts_enabled` column.
    Every helper uses `async with pool.acquire() as conn:` and a single parameterized statement — the
    guild_id/channel_id/enabled values always arrive as `$1`/`$2` params (never string-interpolated), which
    is the confused-deputy / injection guard (T-19-03).
  </action>
