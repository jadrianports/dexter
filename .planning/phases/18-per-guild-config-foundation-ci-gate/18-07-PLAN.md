<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| untrusted PR code → CI runner | A forked PR's code runs in the workflow |
| CI job → ephemeral service container | test DB creds live only inside the job |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-CIPRIV | Elevation of Privilege | GitHub Actions trigger + token scope | mitigate | `pull_request` (never `pull_request_target`) runs a fork's PR with the PR author's own permissions; top-level `permissions: contents: read` denies write even to the default `GITHUB_TOKEN`; zero `secrets.*` referenced, so a malicious PR cannot exfiltrate anything |
| T-18-CICREDS | Information Disclosure | pgvector service-container creds (`POSTGRES_PASSWORD: dexter`) | accept | Ephemeral, network-isolated to the single job, torn down after the run; zero overlap with production Neon `DATABASE_URL` — the literals never touch prod (D-15 "zero secrets" design) |
| T-18-SC | Tampering | `pip install` of runtime + `ruff` in CI | mitigate | Ruff was vetted + pinned (`>=0.15,<0.16`) and approved in 18-01's blocking-human legitimacy gate; runtime deps come from the existing pinned `requirements.txt`; no new package is introduced by this plan |
</threat_model>
