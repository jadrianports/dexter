<objective>
Build the three-workflow topology D-03 locks: an unprivileged **gate** that everything passes through, a privileged **Pages deploy** that only fires behind a green gate on `main`, and a privileged **GHCR publish** that only fires on a `v*` tag.

Purpose: these three concerns need opposite permission levels and opposite triggers, and the project has consistently refused to solve that with an `if:` guard inside one file. `ci.yml` runs on every PR, including from forks — its `permissions: contents: read` ceiling exists precisely to deny an elevated token to untrusted code (T-18-CIPRIV). `actions/deploy-pages` needs `pages: write` + `id-token: write`; GHCR needs `packages: write`. Bolting either into `ci.yml` behind `if: github.ref == 'refs/heads/main'` would mean trusting that nobody ever removes the guard. Three files, three permission ceilings, three triggers.

The accepted cost is that the site builds twice — once to gate, once to deploy. `workflow_run` runs in a different workflow run with a different filesystem; it cannot reach into `ci.yml`'s build output. That is a structural consequence of D-03, not an inefficiency to optimize away.

Output: CICD-02 and CICD-03, both proven by real runs — not by reading YAML.
</objective>

    2. **The `if:` condition needs BOTH filters** — `github.event.workflow_run.conclusion == 'success'` **and** `github.event.workflow_run.head_branch == 'main'`. The `completed` event fires on failure too, and on every branch including PR branches. This is D-12's "the gate must be real": a merge to `main` that breaks the drift guard must not publish a page carrying a wrong invite URL, and a red PR-branch run must not deploy anything at all.
    3. **`actions/checkout` must be passed `ref: ${{ github.event.workflow_run.head_sha }}` explicitly.** Its default checks out the ref of the *currently running* workflow's context, which can differ from the commit CI actually validated if another push landed in between — publishing a page one commit ahead of what was gated (Pitfall 4).
    4. **Rebuild the site here.** A different workflow run means a different filesystem; this job cannot reach `ci.yml`'s already-built `dist/`. That is D-11's accepted "the site builds twice" cost, confirmed structural, not an inefficiency.

    Permissions: `contents: read`, `pages: write`, `id-token: write` — declared in this file, which exists so those three lines never appear in a PR-triggered workflow. Target the `github-pages` environment with `url: ${{ steps.deployment.outputs.page_url }}`. Publish via `actions/upload-pages-artifact` (`path: site/dist`) → `actions/deploy-pages`. **No build output is ever committed** (D-11) — `dist/` stays a pure artifact.

    Do **not** use `withastro/action` here even though it is Astro's official composite action. It bundles install + build + artifact upload, which is fine for this job but wrong for `ci.yml`'s scan job (which must build *without* uploading a Pages artifact, having no `pages: write`). Using plain `setup-node` + `npm ci` + `npm run build` in both files keeps the two builds identical and keeps `dist/`'s path under our control.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| forked-PR code → GitHub Actions runner | `ci.yml` runs on every PR, including from forks. Its `contents: read` ceiling is the control that keeps an untrusted contributor's code away from a write-capable token |
| `ci.yml` (unprivileged) → `pages.yml` (privileged) | The `workflow_run` boundary. A privileged deploy job is reachable **only** via a completed CI run, filtered to success + `main` |
| repo tag → GHCR package registry | A `v*` tag grants `packages: write` and publishes a public-facing artifact |
| Dockerfile build context → published image layers | Anything not excluded by `.dockerignore` becomes a public, permanently-cached layer in a pullable image |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-23-CIPRIV | Elevation of Privilege | `pages: write` / `id-token: write` / `packages: write` reachable from a PR-triggered run | mitigate | D-03's structural split: three files, three ceilings, three triggers. `ci.yml` keeps `contents: read` and is the only PR-triggered workflow; `pages.yml` and `release.yml` are never PR-triggered (asserted by grep). No elevated job is ever guarded by an `if:` a future PR could delete |
| T-23-10 | Elevation of Privilege | `pull_request_target` | mitigate | Asserted absent from all three workflow files. Running a fork's untrusted code with a write/secret-capable token is the exact failure T-18-CIPRIV named |

Forks cannot push to `main`, so the branch filter also closes the fork-triggered-deploy path. Proven by an observed **skipped** deploy job on a throwaway branch |
