# Phase 23 — UI Design Contract

> Visual and interaction contract for the recruiter-facing landing page (`/site`, Astro, zero-JS-by-default) and its two derived surfaces (README case study, README demo GIF). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope note:** this contract governs PORT-01/02/03/04 (the landing page, the demo mock, and the copy/wording that also lands in the README). It does NOT govern CICD-02/03 workflow YAML, which is a plumbing concern the planner owns directly from 23-RESEARCH.md.

## Design System

| Property | Value |
|----------|-------|
| Tool | none — deliberate. shadcn/ui assumes a React runtime + Radix primitives; D-01 locks this page to a zero-JS-by-default Astro static build with no interactive component islands. Introducing a React-oriented component registry into a page whose entire premise is "ship nothing but HTML/CSS" would contradict D-01's stated rationale. Registry safety gate below is therefore **not applicable**. |
| Component library | none — hand-authored `.astro` components (HTML + scoped CSS only, see Component Split). No Radix, no base-ui, no headless-UI kit. |
| Icon library | none — Unicode glyphs only (`↓` `↗` `→`, and a plain 3-dot `···` for the demo's typing indicator, built from `<span>` elements not glyphs). Zero icon font, zero SVG sprite, zero `lucide`/`heroicons`-class dependency. This is a deliberate extension of the zero-JS/zero-third-party posture to the icon layer, and reinforces the terminal aesthetic (terminal UIs lean on text glyphs, not icon fonts). |
| Font | **Headings/labels/CTAs:** `JetBrains Mono` (self-hosted `.woff2`, OFL-1.1 licensed, freely redistributable), weights 400 + 600 only. **Body copy:** system-ui sans-serif stack (`-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, Helvetica, Arial, sans-serif`) — zero download, zero third party. See Typography for the full pairing rationale. |

**Why self-hosted, not Google Fonts CDN:** GitHub Pages + a zero-JS posture means every third-party origin is a cost with no offsetting benefit here — a CDN font request is a render-blocking (or FOIT/FOUT-inducing) cross-origin call the page doesn't need. Self-hosting one mono family (Latin subset only, ~15–20KB per weight as static `.woff2`) keeps the personality-defining typeface under this project's own control and off a third-party request graph entirely. The body font is system-native, so **only the mono family is ever downloaded** — total font payload target: **≤ 45KB** (2 weights × ~20KB).

**Files:** `site/src/assets/fonts/jetbrains-mono-400.woff2`, `site/src/assets/fonts/jetbrains-mono-600.woff2`, declared via `@font-face` in a global stylesheet with `font-display: swap` and preloaded in `Layout.astro`'s `<head>` (`<link rel="preload" as="font" type="font/woff2" crossorigin>` for both weights — this is the one acceptable exception to "zero preload budget" on a page this light).

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|

| sm | 8px | Compact element spacing — button icon-to-label gap, feature card internal gaps |

| 2xl | 48px | Desktop section side-gutter, spacing between hero elements |
| 3xl | 64px | Section-to-section vertical padding (desktop) |

Exceptions:
- **Minimum interactive touch target: 44×44px** on both CTA buttons and all standalone links at the `<640px` breakpoint (WCAG 2.5.5-aligned; not itself a multiple of 4 but the height is achieved via `md`/`lg` padding around mono-13px label text, not a bespoke value).
- **Section vertical padding on mobile is `2xl` (48px), not `3xl`** — `3xl` is desktop-only, to keep the single-scroll page from feeling excessively long on a phone.

## Typography

| Role | Size | Weight | Line Height | Family |
|------|------|--------|-------------|--------|
| Body | 16px | 400 | 1.6 | sans (system-ui stack) |
| Label (meta/timestamps/badges/nav) | 13px | 400 | 1.5 | mono |
| Heading (h2 section titles) | 28px | 600 | 1.2 | mono |
| Display (h1 hero) | 44px | 600 | 1.15 | mono |

**Exactly 2 weights, declared once, used consistently across both families:** 400 (regular) and 600 (semibold). Never introduce a third weight (no 300, no 700, no 800) anywhere on the page.

**Mono-vs-sans pairing decision (Claude's discretion, justified):** headings, labels, timestamps, badges, and both CTA buttons are set in **mono** — this is what makes "terminal-ish" real rather than decorative (D-04). Running **body paragraph copy** (feature blurbs, boundaries prose, hero subhead) is set in **sans** — D-04's hard constraint is that the page "must stay legible" even while being a personality demo, and a full page of monospace prose measurably hurts reading speed at paragraph length. The demo mock's fake chat message text is also set in **sans** (not mono) — Discord's own UI is sans-serif, and using sans here makes the reproduction read as an authentic UI mock rather than a stylized illustration, which matters for D-06's honesty framing (see Demo Mock spec below).

**h3 (feature card titles, boundary item titles):** 18px / 600 / 1.3 / mono — one step below Heading, still mono (titles are short labels, not prose).

**Responsive scale-down (`<640px`):** Display → 32px, Heading → 22px, h3 → 16px. Body/Label sizes are unchanged at every breakpoint (16px body is the accessibility floor; do not shrink it on mobile).

**Verified contrast ratios (against `#0a0d12` background):**
- Text primary `#e7eaf0`: **~15.7:1** — passes AAA for all text sizes.
- Text secondary `#93a0b0`: **~7.3:1** — passes AAA for normal text, comfortably clears AA (4.5:1) even at 13px label size.
- Accent `#ffb454` (used as link/text color): **~11.0:1** — passes AAA.
- Accent-as-button-background with dark text on top (`#0a0d12` text on `#ffb454` fill): same **~11.0:1** ratio inverted — passes AAA for button labels.
- Disclosure marker `#e2793f`: **~6.5:1** — passes AA comfortably; used only for small marker glyphs and a status dot, never for running text, so this exceeds what's required.

**No box-shadows anywhere.** Flat design, 1px borders only — reinforces the terminal aesthetic and keeps the CSS payload minimal.

## Demo Mock Spec (PORT-02 / D-06)

**What it is:** a static-markup, CSS-animated reproduction of a two-exchange Discord conversation, inside a "window chrome" frame (a small top bar showing a fake channel name and three decorative dots, evoking a chat app without copying Discord's actual logo/mascot — see Honesty & Branding Guardrails below).

**Structure (top to bottom):**
1. Window chrome bar: `# general` (mono, Label size, muted) + 3 small decorative dots (pure CSS circles, not an SVG asset).
2. Message 1 — human speaker (fictional handle, see Placeholder Contract).
3. Message 2 — Dexter's reply, preceded by a typing indicator (3 pulsing dots, CSS-only).
4. Message 3 — human speaker, second line.
5. Message 4 — Dexter's reply, preceded by a second typing indicator.
6. Caption below the frame, Text-secondary, Label size: **"real words. reconstructed pixels."** (directly lifted from D-06's own framing — honest, on-voice, does not claim a screen recording).

**Message row anatomy (`DemoMessage.astro`):**
- Avatar: a 40px (32px on `<480px`) circle, solid fill, no image asset. Human speaker avatar = neutral gray (`#3a4250`) with a single-letter initial in Text-secondary. Dexter's avatar = Accent-amber fill with a "D" monogram in dark text (`#0a0d12`) — **not** a copy of any real Discord/Dexter mascot art (none exists to copy from; this is a fresh, simple monogram).
- Username row: username (Label, Text-primary, 600 weight) + a small "BOT" pill badge next to Dexter's username only (radius-sm, Secondary-surface background, Border outline, Label-size text) — a generic, non-trademarked "this is a bot account" convention, not a reproduction of Discord's actual badge asset.
- Timestamp: Label size, Text-secondary, inline after the username on `≥480px`; **hidden below 480px** (progressive disclosure — not essential content, prevents crowding on the narrowest screens; see Responsive Behavior).
- Message bubble: sans body text, 16px/1.6, Text-primary, no bubble background/border (Discord's own default flat message style, not a bubble-chat skin) — this is a deliberate authenticity choice, not a corner-cut.

**Placeholder contract (D-06 hard rule — read before touching this component):**

```ts

// PLACEHOLDER CONTRACT — DO NOT AUTHOR REPLACEMENT TEXT FOR dexterLine FIELDS.
// These two strings must be replaced with VERBATIM, UNEDITED real Dexter output

  { speaker: "dexter", text: "{{DEXTER_DEMO_LINE_1}}" }, // verbatim /ask or /roast output — DO NOT AUTHOR
  { speaker: "human", handle: "wrenlow", text: "playing this on repeat again lol" },
  { speaker: "dexter", text: "{{DEXTER_DEMO_LINE_2}}" }, // verbatim ambient/roast output — DO NOT AUTHOR

```

- `wrenlow` is a **fictional, generic placeholder username** for the human speaker — inventing a plausible handle for demo scaffolding is fine and outside D-06's constraint (D-06 governs Dexter's dialogue only, not the surrounding fake UI chrome). Do not use a real Discord username.
- The two human lines above (`"dex what do you think of my playlist"`, `"playing this on repeat again lol"`) are scaffolding to give the sourced Dexter lines a plausible setup — also fine to author, since they are not attributed to Dexter and don't claim to be captured output.
- If the two `{{DEXTER_DEMO_LINE_*}}` tokens are still present when this phase reaches verification, **PORT-02 is incomplete**, not merely "using placeholder styling" — this is a hard content gate, not a cosmetic one.

**Animation mechanism (CSS keyframes only, per D-01's zero-JS-by-default premise):**

- A single looping keyframe timeline, total duration **16s**, `animation-iteration-count: infinite`, applied to the `DemoMock` container. Each message/indicator element gets a `animation-delay` offset into that shared timeline (staged reveal): message 1 appears at 0.5s, typing indicator 1 shows 2s–4s, message 2 appears at 4s, message 3 appears at 7s, typing indicator 2 shows 9s–11s, message 4 appears at 11s, the full conversation holds fully visible from 13s–16s, then the loop resets (all 4 messages present, then the animation restarts — it should NOT visually "wipe" back to empty; the reset is instantaneous at the keyframe boundary, and the very next thing a looping visitor sees is message 1 re-appearing, i.e. the reset reads as a natural restart, not a glitch).
- **Because the loop is finite and fully repeats, a visitor arriving mid-cycle sees at most 16s of a partial state before the conversation replays from the top** — this satisfies "replayable or loop sensibly; a visitor arriving mid-animation shouldn't see a half-finished joke forever." No JS-driven "restart on scroll into view" is needed given the loop is short and honest about being a loop.
- Implementation: each message element starts at `opacity: 0; transform: translateY(6px)`, animates to `opacity: 1; transform: translateY(0)` over 400ms at its delay offset, then holds. The typing indicator is a separate small element (3 `<span>` dots) with an independent, faster (1.2s) infinite bounce animation, itself wrapped in the same overall visibility/opacity staging so it only renders during its window.

**`prefers-reduced-motion` (HARD REQUIREMENT, not optional):**

```css
@media (prefers-reduced-motion: reduce) {
  .demo-mock * {
    animation: none !important;
    transition: none !important;
    opacity: 1 !important;
    transform: none !important;
  }
}
```

Under reduced motion, **all 4 messages and zero typing indicators render immediately in their final, fully-revealed state** — the demo becomes a static (but still fully readable) transcript. This is the entire content, not a degraded stub — a reduced-motion visitor loses only the timing/reveal effect, never any information.

## Honesty & Branding Guardrails (D-06, D-04 cross-cutting)

- **Never** use Discord's actual logo, mascot, or brand asset anywhere on the page (hero, demo mock, CTA buttons, footer). The caption **"real words. reconstructed pixels."** is the one on-page acknowledgment required, and it is sufficient — no larger disclaimer banner is needed or wanted (D-06 doesn't require a big disclosure box, just no false claim).

If in doubt during implementation, favor the more generic/abstracted version of any UI chrome element over a pixel-accurate Discord copy.

## Invite URL Sourcing (HARD FENCE — Phase 22 D-07)

**The rule:** `logic/invite.py::build_invite_url()` is the ONLY permitted constructor of Dexter's invite URL anywhere in this repo (Phase 22 D-03/D-07). Astro cannot import Python, so this page necessarily contains the one unavoidable second literal — but it must be the **exact same computed string**, sourced from the same public constants, never re-derived or shortened.

**Exact canonical string (computed live this session against the actual `config.py`/`logic/invite.py` at HEAD — copy this literally, do not regenerate it by hand):**

```
https://discord.com/oauth2/authorize?client_id=1492588698364018898&scope=bot+applications.commands&permissions=309240908864
```

**Sourcing mechanism:**

```ts
// site/src/config.ts
// Mirrors config.py's DISCORD_CLIENT_ID / INVITE_PERMISSIONS_VALUE / INVITE_SCOPES
// (see config.py lines ~282-304) and logic/invite.py::build_invite_url()'s output
// shape. If either side ever changes, the D-02 CI drift scan (tests/test_site_drift_guard.py)
// fails the build — that scan is the safety net for this necessary duplication.
const DISCORD_CLIENT_ID = "1492588698364018898";
const INVITE_PERMISSIONS_VALUE = "309240908864";
const INVITE_SCOPES = "bot+applications.commands"; // pre-encoded: discord.py's oauth_url()
                                                     // joins scopes with a space, which
                                                     // application/x-www-form-urlencoded
                                                     // encodes as "+" — this is not a typo.

export const INVITE_URL =
  `https://discord.com/oauth2/authorize?client_id=${DISCORD_CLIENT_ID}` +
  `&scope=${INVITE_SCOPES}` +
  `&permissions=${INVITE_PERMISSIONS_VALUE}`;
```

`Hero.astro` and `Cta.astro` both `import { INVITE_URL } from "../config"` and render `<a href={INVITE_URL} class="cta-button">add to discord</a>` — a **static, build-time-resolved literal string landing in the rendered HTML**, never constructed or mutated by client-side JS (constructing it in JS would place the URL outside the D-02 static `dist/*.html` scan's reach, defeating the guard by construction — do not do this under any circumstance, including "just for the click handler").

**HARD VERIFICATION GATE — a real risk this spec flags, not covered by 23-RESEARCH.md:** Astro's compiler is documented (Context7, `withastro/docs`, "Deprecated Unescaped HTML in Expressions") to HTML-entity-escape text-node expression content (`<h1>{title}</h1>` → entity-encodes `<`/`>`). Whether the same entity-escaping applies to `&` characters **inside an interpolated attribute value** (`href={INVITE_URL}`) is not confirmed by the fetched documentation either way. This matters concretely: the invite URL contains two literal `&` characters, and the D-02 CI scan does a **literal byte-string comparison** — if Astro emits `&amp;` in place of `&` in the built `dist/index.html`, the scan will report a **permanent false-positive drift failure** on every build, because the canonical Python-side string has raw `&` and the rendered HTML would have `&amp;` (the link would still work correctly for a human clicking it — browsers decode `&amp;` back to `&` when parsing an attribute — but the raw source bytes would differ, which is exactly what the scan checks).

**Required verification step before wiring the CI scan (execution-time, not before):**
```bash
cd site && npm run build
grep -F 'https://discord.com/oauth2/authorize?client_id=1492588698364018898&scope=bot+applications.commands&permissions=309240908864' dist/index.html
```
- **If the grep matches** (raw `&`, not `&amp;`): proceed, no further action needed.
- **If it does not match** because the built HTML contains `&amp;` instead: replace the two anchor tags' `href` interpolation with Astro's `set:html` directive on a small raw-HTML string for just that one attribute (e.g. `<Fragment set:html={`<a href="${INVITE_URL}" class="cta-button">add to discord</a>`} />`), which Astro's own docs confirm bypasses escaping entirely. This is a code-review item for whichever plan wires the CTA buttons, not a decision to make blind at design time.

## Responsive Behavior

**Breakpoints (mobile-first):**
- **Base (mobile):** `< 640px`
- **Tablet:** `640px – 1023px`
- **Desktop:** `≥ 1024px`

**General page behavior:**
- Single column at every breakpoint — the layout never becomes multi-column except the Features grid (below).
- Features grid: 1 column `<640px`, 2 columns `640–1023px`, 3 columns `≥1024px`. Boundaries: always single-column, full-width items (4 stacked boundary panels) at every breakpoint — a boundaries grid would visually trivialize disclosure content that D-10 wants read carefully, one at a time.
- Section side gutters: `lg` (24px) `<1024px`, `2xl` (48px) `≥1024px`.
- Both CTA buttons are full-width on `<480px` (easiest tap target on a phone), auto-width (content-sized) at `≥480px`.

**Demo mock — the most fragile element, explicit degradation contract:**
- `≥1024px`: mock renders at a fixed max-width of 560px, centered in the Demo section.
- `640px–1023px`: mock scales to 100% of the section's content width (no fixed max-width), avatars stay at 40px.
- `<640px`: mock is 100% width minus the section's `lg` gutter; avatars shrink to 32px; the window-chrome bar's decorative dots shrink proportionally.
- `<480px`: timestamps are hidden (see Demo Mock Spec) — username + BOT badge remain, message text always wraps normally (never horizontal scroll, never `white-space: nowrap` on message bubble text under any circumstance — this is the one hard rule for the fragile element: **text always wraps, never clips, never scrolls horizontally**).
- The animation timing (16s loop) is identical at every breakpoint — do not speed up or slow down the loop responsively; only geometry changes.

## Accessibility Contract

- **Landmarks:** `<main>` wraps all page content; each major section (`Hero`, `DemoMock`, `Features`, `Boundaries`, `Cta`) is a `<section>` with `aria-labelledby` pointing at that section's heading `id`; `<footer>` for the closing footer. No `<nav>` element exists (there is no navigation menu — a single scroll page with two identical CTAs doesn't need one).
- **Skip link:** a visually-hidden-until-focused "skip to main content" link is the very first focusable element in `Layout.astro`, targeting `#main-content`. Cheap, real accessibility signal on a portfolio page an engineer will scrutinize.
- **Heading hierarchy:** exactly one `<h1>` (Hero). `<h2>` for each of the four remaining section headings (Demo, Features, Boundaries, CTA). `<h3>` for feature card titles and boundary item constraint-labels. No heading level is skipped.
- **Both CTAs are real `<a href>` elements**, never a `<button>` with a JS click handler (there is no JS to handle a click with, and a real link is the correct semantic + keyboard-native element regardless).
- **Focus visibility:** every interactive element (both CTA links, the scroll-to-demo micro-link, the "browse the code" link, footer links) gets a **2px solid Accent-amber outline, 2px offset**, on `:focus-visible` — never suppressed with `outline: none` without a replacement, and the Accent color's 11:1 contrast ratio makes this genuinely visible against the dark background at every location on the page.
- **Contrast:** all text/background pairs verified ≥ 4.5:1 (several ≥ 7:1); see Color table's contrast ratios above. No text is ever rendered at the Border/divider color.
- **Reduced motion:** hard requirement, full CSS block specified under Demo Mock Spec — the ONLY animated element on the page is the demo mock (plus the hero's blinking-cursor decorative glyph, which must independently honor the same `prefers-reduced-motion` query and render as a static, non-blinking cursor character when reduced motion is requested).
- **Decorative fake-UI transcript, announced sensibly (the demo mock's specific a11y pattern):** the mock's DOM must not be exposed to assistive technology as if it were a real, live, interactive chat widget — a screen reader user should not be made to navigate 4 fake "messages" as though they were live content items. Pattern:
  ```html
  <div class="demo-mock" role="img" aria-labelledby="demo-mock-label">
    <p id="demo-mock-label" class="sr-only">
      Animated mock of a Discord conversation demonstrating Dexter's personality.
    </p>
    <div aria-hidden="true">
      <!-- all visual message rows, avatars, typing indicators, chrome bar -->
    </div>
    <p class="sr-only">
      wrenlow: dex what do you think of my playlist.
      Dexter replies: {{DEXTER_DEMO_LINE_1}}.
      wrenlow: playing this on repeat again lol.
      Dexter replies: {{DEXTER_DEMO_LINE_2}}.
    </p>
  </div>
  ```
  The visual markup is `aria-hidden="true"` (decorative), the outer wrapper is announced once as an image-like unit via `role="img"` + `aria-labelledby`, and a visually-hidden (`.sr-only`) plain-text transcript immediately follows so the actual **content** (the joke) — not just the fact that a demo exists — reaches non-visual users. This is the concrete answer to "still needs to be readable/announced sensibly."
- **Alt text:** the demo mock's fake avatars are `aria-hidden="true"` (decorative, covered by the pattern above) — no individual `alt` text needed on them since they carry no information the `.sr-only` transcript doesn't already state.
- **Language:** `<html lang="en">` in `Layout.astro`.
- **`.sr-only` utility class** (standard visually-hidden-but-focusable/announced pattern) is defined once in `global.css` and reused for the skip link and the demo-mock transcript fallback — do not hand-roll a second variant.

## Performance & Weight Budget

- **Zero client-side JS** by default (D-01) — no bundle to budget.
- **Font payload:** ≤ 45KB total (2 self-hosted `.woff2` weights, Latin subset).
- **No raster images** on the page — avatars are CSS circles, the window-chrome dots are CSS circles, all icons are Unicode glyphs. Only asset beyond fonts + CSS is the favicon (`site/src/assets/favicon.svg` or `.ico`, small).
- **Total initial page weight target (excluding font, which loads async via `swap`): well under 50KB of HTML+CSS.** This is achievable given the component/asset inventory above — flag in review if it balloons past that without a clear reason.
