<objective>
Stand up the Astro project (D-01), wire the invite URL through the one legal path (Phase 22 D-07), and — most importantly — **rebuild the drift guard so it still guards something** (D-02).

    Collect files with a filesystem walk — `_repo_root() / "site" / "dist"` then `rglob("*.html")` — **not** `git ls-files`. Derive the path from `_repo_root()` rather than a bare relative `Path("site/dist")` so the test is cwd-independent (CI runs pytest from the repo root; a local run may not).

    1. **The guard** — collect every `dist/**/*.html`, run them through `_collect_offenders(files, _canonical_url())`, assert zero offenders. Handle the empty case with the **skip/fail split** that is the entire point of D-02: if no files are found and `SITE_DIST_REQUIRED` is set in the environment, call `pytest.fail()` with a message saying the Astro build step did not run or produced no output; if it is not set, `pytest.skip()` with a message saying the site isn't built locally. That does not apply here (Astro ships zero JS by default per D-01, and both CTAs are static anchors per D-05), but the boundary of the guarantee gets written down rather than glossed.

IN-01 was the concern that the source-file guard's `.md`/`.html`/`.txt` extension allowlist would miss a URL in a new file type. Widening the allowlist to `.astro`/`.ts` would **not** fix it and would actively make things worse: `site/src/config.ts` assembles the URL from concatenated template literals, so the full canonical string never appears contiguously — the regex would match the fragment `https://discord.com/oauth2/authorize?client_id=${DISCORD_CLIENT_ID}` and report it as permanent, unfixable drift. The correct fix is a different collection strategy over the *rendered output*, where the URL is a literal regardless of how the template built it. That is a strictly stronger guarantee than the allowlist widening would have been. Say so, in the docstring, in those terms.
