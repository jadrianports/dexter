This plan also makes the two open questions and the flagged anchor discrepancy CONCRETE in code:
- **OQ1 (kind-scoped recall):** add `kind: str | None = None` to `database.search_memories` and
  `MemoryService.recall`; append `AND kind = $N` to the SQL ONLY when provided (never `kind IS NULL`),
  so every existing Phase 11/15 call site stays byte-identical.
- **OQ2 (co-occurrence definition):** `get_artist_cooccurrence` = same-guild-calendar-day bucket join
  over `song_history`, bounded by a recency window, computed entirely in SQL (guild-wide aggregate,
  no per-user attribution).
- **Anchor discrepancy (D-04 vs. schema):** DECIDED **Option B** — derive the `/discover` anchor from a
  new guild-scoped `get_user_top_artist(song_history WHERE guild_id AND user_id)` helper, NOT from
  `user_artist_counts` (which has NO `guild_id` column and would surface a cross-server-flavored
  "top artist" — the Pitfall 2 UX surprise). Rationale below in Task 2.

Purpose: give the wave-2 cog plans a fully-tested data layer to consume, so Discord glue never
has to invent SQL or scoping.
Output: 3 new aggregate helpers, a `kind` param on 2 functions, 6 config knobs, and their tests.
