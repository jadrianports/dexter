<objective>
Build the D-14 mixing half in `services/audio.py`: `TruncatingSource` (exhausts the outgoing track
early so discord.py sees an ordinary natural end), `CrossfadeSource` (mixes the outgoing tail into
the incoming head with `audioop`, then continues alone), and the additive `crossfade_from=` kwarg
on `get_source` that is **byte-identical to today when omitted**.

Purpose: this is the shape that makes D-01 satisfiable. The entire fade lives *inside the incoming
track's own `AudioSource`*, so the voice client still only ever sees **one source and one `play()`
per track** — the two live FFmpeg decoders exist only inside that single object, where one
`cleanup()` owns both. RESEARCH proved this against the real engine: attack 2 tore down two live
decoders at both teardown sites with **zero** per-site edits and `ffmpeg` back to 0.
Output: two `AudioSource` subclasses, one additive kwarg, five test guards.

  <read_first>
    - `services/audio.py` lines 179-185 — the "one failing teardown must not abort the rest"
      `try/except OSError: continue` precedent in `cleanup_cache`. `CrossfadeSource.cleanup()`

      0.0019%), and §Round 2 Task 4 (equal-power: `g_out² + g_in² = 1`).
    - `27-CONTEXT.md` "Claude's Discretion" — the curve is the planner's call; equal-power is
      chosen and was the curve the user approved at the D-04 gate.
  </read_first>
  <behavior>
    - `is_opus()` returns `False` (mixing requires decoded PCM).
    - For the first `fade_frames` reads, output == equal-power mix of tail and head.
    - After `fade_frames`, output == the head alone, and the tail has been cleaned and dropped.
    - An empty tail on the very first read degrades to a plain fade-in and never raises.
    - `cleanup()` cleans **both** children even when the tail's cleanup raises.
  </behavior>
  <action>
    Add `class CrossfadeSource(discord.AudioSource)` to `services/audio.py`, constructed as
    `CrossfadeSource(tail, head, fade_frames: int)` where both children are `discord.FFmpegPCMAudio`.

    - `is_opus()`: `return False`. Mixing requires decoded PCM — this is the one place the opus
      fast path is genuinely forfeited, and only for the incoming track's fade window.
    - `read()`: while `self._frames_mixed < self._fade_frames`, read one frame from each child and
      mix with **equal-power** gains: `progress = self._frames_mixed / self._fade_frames`;
      `g_out = math.cos(progress * math.pi / 2)`; `g_in = math.sin(progress * math.pi / 2)`.
      Combine via `audioop.add(audioop.mul(tail, 2, g_out), audioop.mul(head, 2, g_in), 2)` —
      width 2 (s16). **No headroom gain** — RESEARCH measured 0.0019% clipping and `audioop.add`
      **saturates** rather than wrapping, so the worst case is negligible clipping, never
      catastrophic distortion. Once `_frames_mixed >= _fade_frames`, clean and drop the tail
      (do not hold it for the rest of the track — RESEARCH §Findings 4) and return the head's
      frames alone.
    - **Tolerate an empty tail.** `FFmpegPCMAudio` returns exactly 3840 bytes or `b""`, so
      `audioop.add`'s equal-length requirement is satisfied without padding — but a short file or
      an `-ss` past EOF yields `b""` on the first read. When the tail is empty, skip the mix and
      emit `audioop.mul(head, 2, g_in)` — i.e. degrade to a plain fade-in, never raise
      (RESEARCH §Findings 5).
    - **`cleanup()` must clean BOTH children, head in a `finally`** so a raising tail cleanup
      cannot strand the head decoder. This is Critical Rule 3's whole surface for this feature:
      the two-decoder window is expressible as one `AudioSource`, so one `cleanup()` owns both and
      **no existing teardown site needs editing**. `cleanup()` must be idempotent — it is called
      once by discord.py's player and possibly again by `_play_track`'s failure paths.

    `import audioop` and `import math` at module top. **No numpy** (D-02). Note in the module
    docstring or beside the import that `audioop` is stdlib on the pinned 3.11 (`Dockerfile:4`,
    `ci.yml`), emits a `DeprecationWarning` on the maintainer's local 3.12, and is removed in 3.13 —
    D-02's accepted cost.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cache file (`data/cache/{video_id}.opus`) → `FFmpegPCMAudio` | The file's *content* originates from YouTube (untrusted upstream) and is decoded by FFmpeg in a subprocess. Pre-existing boundary — crossfade adds a second concurrent decoder over the same trust level, not a new trust level. |
| `Track.video_id` → `cache_path()` → an `-ss` FFmpeg argument | `video_id` is attacker-influenceable in principle (it names a file path and now also lands adjacent to a shell-ish options string). |
| Decoded PCM → `audioop.add`/`mul` | Untrusted audio bytes reach in-process C arithmetic. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-27-09 | Denial of Service | **Two live FFmpeg decoders per transition** | mitigate | Bounded by construction: the tail decoder is cleaned and dropped the instant `_frames_mixed >= _fade_frames` (~4s), and both children are owned by one `cleanup()`. The window is 4s per transition, not per track. RESEARCH attack 2 measured `ffmpeg` 2 → **0** at both teardown sites. Additionally bounded by D-08b (off by default) and the `NOT_CACHED` rung (a fade never spawns a *network* decoder). |
| T-27-10 | Denial of Service | `CrossfadeSource` never terminating / leaking the tail | mitigate | `read()` drops the tail at a fixed frame count; the head drives EOF. `cleanup()` is idempotent so a double-call from `_play_track`'s failure path cannot raise. |
| T-27-11 | Tampering | Orphaned FFmpeg process from a raising child cleanup (**Critical Rule 3**) | mitigate | Head cleaned in a `finally`. Row 13's test drives a **raising** tail and asserts the head still got cleaned — the test is the control, not the code comment. |
| T-27-12 | Tampering | `video_id` / `cut_seconds` interpolated into `before_options` | mitigate | `cut_seconds` is a float derived from `TruncatingSource.frames_read` (in-process, never user input, never metadata). `video_id` reaches only `cache_path()`, which is `self.cache_dir / f"{video_id}.opus"` — a fixed-extension join, and the file must already `exists()` before the branch is taken. discord.py passes FFmpeg args as a list (no shell), so no injection vector exists. Unchanged from the existing seek path. |
| T-27-13 | Denial of Service | `audioop.add` integer overflow on loud material | accept | RESEARCH verified `audioop.add` **saturates** (`30000+30000 → 32767`), it does not wrap. Worst case is 0.0019% clipping — measured, on real music, with equal-power and no headroom gain. |
| T-27-14 | Information Disclosure | The D-10b hard-cut log line | accept | Logs a `FadeVerdict` value string (a fixed internal literal). No track title, user id, or path is interpolated. |
| T-27-SC | Tampering | npm/pip/cargo installs | mitigate | **Zero installs this plan.** `audioop` and `math` are stdlib on the pinned 3.11; **numpy is rejected** (D-02) and `grep -c numpy services/audio.py == 0` is an acceptance criterion. No `requirements.txt` change → no package-legitimacy checkpoint required. |
</threat_model>
