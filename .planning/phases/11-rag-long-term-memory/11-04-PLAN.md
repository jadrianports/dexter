<objective>
Build the storage half of the write path: `MemoryService.remember(fact_text, kind, salience)` — embed
the distilled fact (RETRIEVAL_DOCUMENT, priority 2 background), dedup against the user's nearest
existing memory (bump `hit_count` when near-identical, else insert with an `expires_at` decay horizon),
then enforce the per-user cap by evicting the lowest-value memory. Add the pure dedup/salience/eviction
functions and the DB write helpers.

This plan owns the STORAGE + DEDUP + CAP mechanics operating on an ALREADY-DISTILLED fact. The
distillation + sensitivity gate that PRODUCES the fact, and the triggers that CALL remember(), are
11-05 (interface-first: define the storage contract here, feed it next).

Purpose: MEM-04 (write-time near-dup dedup) + MEM-07 (per-user cap eviction on salience, D-07/D-08).

Output: `remember()` that never inserts a near-duplicate, keeps each user bounded at
MEMORY_MAX_PER_USER, and degrades silently on embed rate-limit (priority-2 background).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| distilled fact → remember() | trusted-but-bounded text gets embedded + stored durably |
| remember() → Neon | per-user write + scoped eviction |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-04a | Denial of Service | unbounded per-user memory growth (#1 companion-bot failure) | mitigate | count > MEMORY_MAX_PER_USER triggers evict_lowest_salience on every insert (Pitfall 13/14); decay sweep in 11-07 backstops |
| T-11-04b | Tampering | SQL injection via fact/embedding fields | mitigate | $N parameterized asyncpg; embedding via register_vector typed list — never string-built SQL |
| T-11-04c | Information Disclosure | eviction deletes another user's rows | mitigate | evict_lowest_salience is scoped `WHERE user_id = $1 AND id = ANY($2)` |
| T-11-04d | Denial of Service | embed write saturates the embedding quota | mitigate | writes are priority-2 (reject-if-wait>10s) on the separate _embed_limiter; saturation skips the write |
</threat_model>
