<objective>
Self-heal transient YouTube search/extract failures within a bounded budget (REL-06 / D-08).

Today `async_search` and `async_extract` run `self.search` / `self.extract` in an executor and
propagate the first failure. `download()` already self-heals (throttled yt-dlp update + retry).
This plan brings search/extract up to the same standard: a quick bounded retry first (most blips
are transient network errors), then ‚ÄĒ only if that still fails ‚ÄĒ the existing throttled
`update_ytdlp()` + final attempt. A permanent failure (`ExtractorError.expected=True`: video
unavailable, age-restricted) bypasses both, so we never burn retries/updates on content that will
never load.

Purpose: a single network blip during a `/play` search currently fails the whole command; bounded
retry recovers it transparently, and the update fallback still catches a real yt-dlp breakage the
way download() does ‚ÄĒ without paying the slow update on every blip.
Output: resilient `async_search` / `async_extract` + a pure transient-vs-permanent classifier with
unit tests.
</objective>
